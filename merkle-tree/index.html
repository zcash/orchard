The Note Commitment Tree

1. Why This Chapter Exists​

Orchard's anti-double-spend is a single global Merkle tree of all note commitments. A spender proves, in zero-knowledge, that their input note is in this tree at a specific position. After this chapter the reader can produce the empty roots by hand, trace combine_inner to the Sinsemilla call, and explain the role of incrementalmerkletree.

The lazy_static! blocks precompute UNCOMMITTED_ORCHARD and the 33-entry EMPTY_ROOTS table once per process.

3.2 MerkleHashOrchard​

A pallas::Base wrapper that implements incrementalmerkletree::Hashable. The trait's combine is forwarded to combine_inner, which performs the Sinsemilla call of Definition 2.2.

3.3 Frontier Maintenance​

The Orchard tree itself is not stored in this crate; the incrementalmerkletree crate maintains a frontier (the right-most authentication path plus the right-most leaf at each level), which is enough to append new leaves and produce paths for live notes.

3.4 Inside the Circuit​

The Merkle path is hashed inside the Action circuit by the MerkleChip from halo2_gadgets; the chip recomputes the chain of Sinsemilla calls and constrains the final xxx-coordinate to equal the public anchor.

4. Failure Modes​

  • Off-by-one depth. MERKLE_DEPTH_ORCHARD = 32 means 32 levels of inner hashes; PRs that introduce 33 or 31 in any hash level off-set break consensus.
  • Forgetting ℓ\ellℓ in the input. The depth tag prevents a collision between hashes at different levels. Removing it lets a path at level 5 be reinterpreted as a path at level 6.
  • Uncommitted collision. The value 222 was chosen because no genuine note commitment is exactly 222. Adding a new "uncommitted" sentinel without checking the collision argument is a soundness risk.

  1. Compute EMPTY_ROOTS[0..3] by hand (running Sinsemilla) and cross-check against the lazy-static table by writing a one-line unit test.
