# qa-bot

Local RAG chatbot over the onboarding docs, complementary to the
Docusaurus full-text search. Embeds every chunk of `onboarding/docs/`
with a small CPU embedding model, stores them in SQLite via
`sqlite-vec`, and at query time retrieves the top-k chunks and hands
them to an OpenAI-compatible LLM with a strict prompt that requires
inline citations or refusal.

This tool is intentionally tiny and generic: the same code drops into
any course generated by the `onboard-repo` skill by overriding three
environment variables (`QA_BOT_DOCS_DIR`, `QA_BOT_SITE_BASE`,
`QA_BOT_DB_PATH`).

## What it ships

- `qa-bot index` – walk `onboarding/docs/`, chunk by H2 / H3, embed
  with `BAAI/bge-small-en-v1.5` on CPU, persist to SQLite.
- `qa-bot retrieve "<query>"` – show the top-k retrieved chunks
  without calling an LLM. Useful for tuning chunking and inspecting
  what the retriever sees.
- `qa-bot ask "<query>"` – full RAG: retrieve, build the prompt, call
  the configured LLM, print the answer plus the sources.
- `qa-bot serve` – FastAPI server on `127.0.0.1:8765` with a vanilla
  HTML chat UI.
- `qa-bot info` – show resolved config (paths, models, whether the
  API key is set, how many chunks are indexed).

## Getting started

The tool uses [`uv`](https://github.com/astral-sh/uv) for Python
project management. Install `uv` first.

```bash
cd onboarding/tools/qa-bot

# Install Python dependencies (creates .venv/, downloads packages).
uv sync

# Configure the LLM backend. The defaults target OpenRouter; copy
# .env.example to .env and edit.
cp .env.example .env
$EDITOR .env

# First-time index build. Also downloads the embedding model
# (~130 MB) into fastembed's cache directory.
uv run qa-bot index

# CLI Q&A.
uv run qa-bot ask "What is rho in an Orchard note?"

# Or run the server and open http://127.0.0.1:8765.
uv run qa-bot serve
```

Equivalent Makefile targets are exposed from the parent
`onboarding/` directory: `make qa-bot-install`, `make qa-bot-index`,
`make qa-bot-serve`, `make qa-bot-ask QUESTION="..."`,
`make qa-bot-info`.

## Configuration

Everything is read once at process start from environment variables;
local development reads them from `.env` via `python-dotenv`.

| Variable             | Default                                          | Purpose                                                          |
| -------------------- | ------------------------------------------------ | ---------------------------------------------------------------- |
| `QA_BOT_API_KEY`     | (unset)                                          | Required for `ask` / `serve`. Bearer token sent to the LLM.      |
| `QA_BOT_BASE_URL`    | `https://openrouter.ai/api/v1`                   | OpenAI-compatible base URL. Point at a local server to go offline.|
| `QA_BOT_MODEL`       | `meta-llama/llama-3.3-70b-instruct`              | Model identifier the backend understands.                        |
| `QA_BOT_EMBED_MODEL` | `BAAI/bge-small-en-v1.5`                         | fastembed model name. Change `QA_BOT_EMBED_DIM` to match.        |
| `QA_BOT_EMBED_DIM`   | `384`                                            | Embedding dimensionality of the chosen model.                    |
| `QA_BOT_DOCS_DIR`    | `<repo>/onboarding/docs`                         | Where to find Markdown to index.                                 |
| `QA_BOT_DB_PATH`     | `<repo>/onboarding/tools/qa-bot/data/qa-bot.db`  | SQLite database holding chunks and embeddings.                   |
| `QA_BOT_SITE_BASE`   | `https://dannywillems.github.io/orchard`         | URL prefix used in source citations.                             |

## Going fully local (no OpenRouter)

The bot speaks the OpenAI chat-completions schema, so any local
inference server works. Two common ones:

- **Ollama (>= 0.1.39).** Start the server, pull a model, point the
  bot at it:
  ```bash
  ollama serve &
  ollama pull qwen2.5:14b-instruct-q5_K_M
  export QA_BOT_BASE_URL=http://localhost:11434/v1
  export QA_BOT_MODEL=qwen2.5:14b-instruct-q5_K_M
  export QA_BOT_API_KEY=ollama  # any non-empty placeholder
  ```
- **LM Studio.** Open the GUI, load a model, start the local server
  (port 1234 by default), then:
  ```bash
  export QA_BOT_BASE_URL=http://localhost:1234/v1
  export QA_BOT_MODEL=<the model name LM Studio shows>
  export QA_BOT_API_KEY=lm-studio
  ```

## Trade-offs and limits

- **Quality is bounded by the LLM you point it at.** A 7B local
  model produces confidently wrong answers about cryptography. Test
  before exposing this to anyone else, and prefer a stronger model
  (hosted, or 14B+ local with patience) for technical content.
- **Embedding quality matters too.** `bge-small` is a budget choice
  for a tool that runs on a laptop. For deeper retrieval accuracy on
  math-heavy content, swap to `BAAI/bge-base-en-v1.5` (768 dim,
  ~440 MB) or `BAAI/bge-m3` (1024 dim, ~2.2 GB) and rebuild the
  index.
- **The corpus is small.** SQLite scans the vector table linearly;
  on a doc site this is fine. For tens of thousands of chunks,
  swap in HNSW via `vss0` or a real vector DB.
- **No streaming.** Answers arrive in one shot. Adding streaming
  needs an SSE endpoint and a small UI tweak; not done yet.
- **No rate limiting or auth.** Bind to `127.0.0.1` only when
  running locally. Public deployments need a reverse proxy with
  rate limits and probably a captcha.

## Architecture

```
+--------------------+      embed       +-----------------------+
| onboarding/docs/*  | --------------> | sqlite + sqlite-vec   |
| (Markdown source)  |  fastembed CPU   | chunks + embeddings   |
+--------------------+                  +-----------+-----------+
                                                    |
                                                    v
+--------------------+     embed q     +------------+----------+
|   user question    | --------------> |    retrieve top-k     |
+--------------------+                  +------------+----------+
                                                    |
                                                    v
                                       +------------+----------+
                                       | prompt + chat call    |
                                       | (OpenAI-compatible)   |
                                       +------------+----------+
                                                    |
                                                    v
                                       +------------+----------+
                                       | answer + sources      |
                                       +------------------------+
```

## How it complements the Docusaurus search

The site's existing full-text search (Lunr via
`@easyops-cn/docusaurus-search-local`) is exact-match-oriented: it
finds pages whose content contains the searched tokens. It is fast,
no infrastructure required, and obviously correct. It is also poor
at answering questions phrased in natural language ("Why does
Orchard have a `rho` field while Sapling does not?").

The qa-bot is the inverse: vector retrieval surfaces semantically
related chunks even when the query shares no tokens with them, and
the LLM stitches the chunks into an answer with citations. It is
slow, costs API tokens, and can hallucinate. Use both: keyword
search for known-token lookups, qa-bot for natural-language
questions.
