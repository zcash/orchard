RedPallas, the Action Signature Scheme

1. Why This Chapter Exists​

Orchard signs two things per bundle: the binding commitment with the binding signature, and each Action's rk with a spend-authorising signature. Both use RedDSA over Pallas (RedPallas). The re-randomisable property lets each Action sign under a one-time public key without revealing the wallet's ak\mathsf{ak}ak. After this chapter the reader understands the difference between the two flavours and can derive their bases.

2. Definitions​

Definition 2.1 (RedDSA Sign)​

Over a prime-order group with base G\mathcal{G}G and order rrr, with hash H:{0,1}∗→FrH : \{0, 1\}^* \to \mathbb{F}_rH:{0,1}∗→Fr​, to sign mmm under secret sk\mathsf{sk}sk with public pk=[sk]G\mathsf{pk} = [\mathsf{sk}] \mathcal{G}pk=[sk]G:

  1. Sample nonce r∈Frr \in \mathbb{F}_rr∈Fr​ deterministically from sk\mathsf{sk}sk and mmm.
  2. R=[r] GR = [r]\, \mathcal{G}R=[r]G.
  3. c=H(R∥pk∥m)c = H(R \mathbin{\|} \mathsf{pk} \mathbin{\|} m)c=H(R∥pk∥m).
  4. s=r+c⋅sk(modr)s = r + c \cdot \mathsf{sk} \pmod rs=r+c⋅sk(modr).
  5. Output σ=(R,s)\sigma = (R, s)σ=(R,s).

Definition 2.2 (Re-randomisation)​

For randomiser α∈Fr\alpha \in \mathbb{F}_rα∈Fr​, the randomised key is rk=pk+[α] G\mathsf{rk} = \mathsf{pk} + [\alpha]\, \mathcal{G}rk=pk+[α]G. The randomised secret is sk+α\mathsf{sk} + \alphask+α, and a signer with both can sign messages under rk\mathsf{rk}rk.

Definition 2.3 (Orchard Flavours)​

  • SpendAuth: base Gak\mathcal{G}_{\mathsf{ak}}Gak​, signs the per-Action SIGHASH; key is rk\mathsf{rk}rk.
  • Binding: base R\mathcal{R}R (the value commitment randomness base), signs the bundle-level SIGHASH; key is bvk\mathsf{bvk}bvk from Chapter 13.

4. Failure Modes​

  • Identity rk. #492 added an explicit rejection of identity rk in Action::from_parts. The defence catches a malformed bundle that would otherwise pass verification under a special case of the Schnorr equation.
  • Nonce reuse. RedDSA derives the nonce deterministically from sk\mathsf{sk}sk and the message; a buggy override that resamples randomly opens the door to a discrete-log recovery attack on sk\mathsf{sk}sk.
  • Wrong base. Signing a SpendAuth message with the Binding base or vice versa produces a verifier that accepts unrelated values. The marker types prevent this at compile time; do not bypass them.
  • alpha = 0. If α=0\alpha = 0α=0, rk=ak\mathsf{rk} = \mathsf{ak}rk=ak and unlinkability is lost. The builder must sample α≠0\alpha \neq 0α=0; an override that allows zero is a privacy bug.
